# Build Log

ACL Education Portal frontend. SolidStart 1.x (SSR), TypeScript, Tailwind v4.

- Phase 0: SolidStart SSR scaffold (1.3.2), Tailwind v4 tokens as swappable CSS variables, app shell (header/footer/meta), typed API client with VITE_USE_MOCK fixture layer, Seo + Breadcrumbs helpers. Build clean; dev server renders server-side HTML with canonical and JSON-LD. ✓
- Phase 1: Full file-based route skeleton with `query`+`createAsync` server loaders for every page type (home, stream, course, listing, exam, college + 4 sub-routes, search, legal, 404). slug-id parsing, top-level dispatcher (stream vs `-course`), exam `-exam` suffix matching. Catch-all returns real HTTP 404; ErrorBoundary degrades thrown API errors to NotFound. Typecheck + build clean; all routes verified to return SSR HTML with correct canonicals. ✓
- Phase 2: Homepage. Hero with scoped primary search (all/colleges/courses/exams) and prominent stream entry cards; browse-by-stream and browse-by-popular-city grids; data-driven top-colleges and popular-courses modules; trust band with live counts and the AAJneeti disclosure; global "Get free admission guidance" capture via LeadTrigger + Modal (lead form stubbed, wired in Phase 6). Home data composed from contract endpoints (homeQuery). SSR renders every module; responsive layouts; no em dashes. ✓
- Phase 3: Primary course-x-city listing page. Data-driven H1 "[Course] Colleges in [City]", intro and stats (count, fee range, popular courses). Filter rail (specialisation, sort, college type, exam accepted, approval, fee min/max) that writes every change to the URL query string and reloads via the SSR loader; sticky on desktop, collapsible on mobile. Result cards with URL-driven pagination (mock slices per page). Page-level lead form ("Get details and compare [course] colleges in [city]", LeadForm stub until Phase 6, never "apply to college X"). Supporting "things to know" copy, FAQ accordion (native details/summary, no-JS friendly) with FAQPage JSON-LD, and related city/course internal links. Verified via SSR that type/page/sort change the result set and FAQPage + BreadcrumbList schema and a base-path canonical are present. Typecheck + build clean. ✓
- Phase 4: College detail block stack. Hero (logo, rating, review count, approvals), sticky in-page anchor/tab nav, and the full stacked blocks rendered from one collegeQuery call: Overview (highlights, campus, website), Courses & Fees (expandable per course via details), Admissions (process, eligibility, accepted exams, dates), Placements (table + recruiters), Rankings & Ratings (table), Cutoffs (table), Gallery (lazy images), Reviews & Q&A (v1 placeholder), News (placeholder), Contact & Location (address, optional map embed, per-page guidance lead form). The three sub-routes (/courses-fees, /placements, /admission) plus /reviews render the same stack with their own canonical and unique title and scroll to the matching section on the client; in-page anchors (#placements etc.) deep-link too. operator_disclosure rendered; no manufactured urgency. JSON-LD: BreadcrumbList, CollegeOrUniversity (with AggregateRating + PostalAddress) and one Course per programme. Typecheck + build clean; SSR verified all 10 sections, schema and sub-route canonicals. ✓
- Phase 5: Course and exam info pages plus unified search. Course page already carried definition, eligibility, duration, specialisations, career scope, fees, related exams (now linked) and top colleges with Course JSON-LD; added a consent-ready guidance lead form. Exam page carries overview, eligibility, pattern, dates and syllabus with accepting colleges; added a consent-ready lead form. Search page rebuilt with a type-ahead SearchAutocomplete (debounced suggestions from the search server fn, grouped by colleges/courses/exams, keyboard and mouse select) that is a GET form to /search so it degrades gracefully without JS, plus scope filter tabs (all/colleges/courses/exams) reflected into the URL. Search stays noindex. Verified via SSR that results render server-side, the scope filter hides groups in the rendered HTML, and empty states show. Typecheck + build clean. ✓
- Phase 6: Reusable lead form (consent + OTP). One LeadForm used inline on listing, college, course and exam pages and in a modal on the home page. Fields: name, mobile, email, city, course/stream of interest, current qualification, intended intake year. Consent is a single unbundled checkbox, NOT pre-ticked, with the plain-language CONSENT_TEXT and a Privacy Policy link; consent.checked and CONSENT_TEXT_VERSION are sent with every submit. OTP gate: send OTP to mobile, verify, carry the returned token as otp_token into the lead; submit stays disabled until consent AND verified mobile AND required fields. Anti-spam: off-screen honeypot (hp_field, also dropped server-side), a 20s sessionStorage rate guard, and disabled-until-ready submit. source_page passed per page; UTM params captured from the URL and persisted in sessionStorage. Lead calls route through server actions ("use server") to keep the API base off the client. Success and error states. Verified via SSR that consent is unchecked, honeypot present and submit disabled by default; mock OTP/submit happy path and honeypot drop confirmed. Typecheck + build clean. ✓
